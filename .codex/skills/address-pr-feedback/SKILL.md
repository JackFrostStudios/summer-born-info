# Address PR Feedback

Use this skill to work through PR feedback without overlapping comments, mixed reasoning, or half-validated fixes.

This skill has strict execution rules:

- MUST retrieve the active PR comments before starting implementation work.
- MUST process exactly one PR comment at a time.
- MUST NOT begin processing the next comment until the current in-flight comment is fully resolved.
- MUST use a dedicated sub-agent for the active comment's analysis and implementation.
- MUST require the sub-agent to either:
  - implement a fix, or
  - return a reason why the current implementation is already optimal.
- MUST require the sub-agent to ensure the build and relevant tests pass before returning any fix.
- MUST review a returned fix locally before treating the comment as resolved.
- MUST add a PR response comment with either the fix applied or the reasoning for not making any changes
- MUST add the suffix  `Generated by AI: Model` to any comments added to PR.

## Workflow

1. Identify the target PR and retrieve all unresolved review comments and discussion threads.
2. Order the comments deterministically and select the first unresolved comment as the only active item.
3. Start one sub-agent for that comment and give it only the context needed for that specific feedback item.
4. Wait for the sub-agent to return either a validated fix or a rationale for keeping the current implementation.
5. If the sub-agent returned a fix, review the changes and confirm the comment is resolved, post that fix back to the PR with the required AI tag before moving on.
6. Commit the fix to source control.
7. If the sub-agent returned a rationale, post that rationale back to the PR with the required AI tag before moving on.
8. Only after the active comment is fully resolved, pick the next unresolved comment and repeat.

## Hard Rules

- MUST keep the main thread focused on orchestration, review, validation follow-up, and PR replies.
- MUST NOT process multiple PR comments in parallel.
- MUST NOT start a second implementation sub-agent while another PR comment is still active.
- MUST NOT collapse several comments into one combined implementation task.
- MUST treat "resolved" as one of:
  - code changes reviewed and validated, or
  - a rationale comment posted back to the PR with the required AI tag.
- MUST reopen investigation if the returned fix does not clearly satisfy the comment or if validation is incomplete.
- MUST commit the each fix as a separate self contained commit.

## Retrieve The Comments

Before delegating anything:

- Retrieve unresolved review comments, line comments, and open discussion threads for the target PR.
- Preserve enough metadata to revisit the exact feedback item:
  - comment id or thread id,
  - file and line reference if present,
  - author,
  - raw comment text.
- Prefer the repository's normal PR tooling. If multiple sources are available, prefer the one that exposes unresolved threads cleanly.
- Build a working queue, but activate only the first unresolved comment.

If comment ordering is ambiguous, use a stable order such as review submission time followed by thread position.

## Delegate One Comment At A Time

For the active comment, start a fresh sub-agent and give it:

- the exact PR comment text,
- the relevant file or diff context,
- any repository-specific constraints,
- the instruction that it is working on only this single comment.

Require the sub-agent to do all of the following:

- analyze the specific PR comment,
- decide whether a change is required,
- implement the fix if needed, otherwise explain why the current implementation is optimal,
- run the build and relevant automated tests after any fix,
- report what changed, what validation passed, and any residual risk.

Do not let the sub-agent pick up a second PR comment in the same run.

## Review Returned Fixes

If the sub-agent returns a fix:

1. Inspect the changed files and verify the implementation addresses the exact feedback item.
2. Confirm the build and relevant tests passed, rerunning validation locally if needed.
3. Check for regressions, partial fixes, or misread feedback.
4. Add a response to the PR comment only when the returned change actually satisfies the reviewer concern.
5. Create a small, self contained commit.

If the fix is incomplete, keep the same PR comment active and send a targeted follow-up to another sub-agent or finish the small gap locally. Do not advance the queue until the current comment is truly resolved.

## Reply When No Change Is Needed

If the sub-agent concludes the current implementation is already optimal:

1. Review the rationale for correctness and tone.
2. Post a reply on the PR for that specific comment or thread.
3. Suffix the reply with `Generated by AI: Model`.
4. Explain clearly why no code change is being made.

Do not move to the next comment until that reply is posted.

## Sub-Agent Prompt Shape

Use a prompt structure close to this:

```text
Address PR comment <comment id> for <PR reference>.

Active comment:
<paste the exact reviewer comment>

Relevant context:
<files, diff excerpt, constraints, expectations>

Requirements:
- Work on this comment only.
- Analyze whether the comment requires a code change.
- If a change is required, implement it and ensure the build and relevant tests pass before returning.
- If no change is required, return a concise explanation of why the current implementation is already optimal.
- Return a concise summary of changes or rationale, validation performed, and any residual risks.
```

## Quality Bar

- Keep comment handling sequential and explicit.
- Prefer small, reviewable fixes tied to one feedback item.
- Ensure small, self contained commits to ensure review is easier.
- Require evidence that validation passed before accepting a fix.
- Keep PR replies concise, specific, and professional.
- Never leave an unresolved active comment while starting another one.
